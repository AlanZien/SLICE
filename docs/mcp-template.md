# MCP template — internal reference

This document describes the Handlebars templates that make up an MCP server
generated by SLICE. Every file emitted by `generateMcp(req)` comes from one
template in `src/server/templates/`, rendered against a single context object
built by `buildContext()` in `src/server/services/mcp-generator.ts`.

## Pipeline

```
GenerateRequest
  ↓ buildContext()
TemplateContext  ──┐
                   ├──▶ Handlebars.compile(<template>) ──▶ GeneratedFile
STATIC_TEMPLATES ──┘
```

- The context is **pure data**: no functions, no references to the request object.
- Templates are compiled once and cached (`templateCache`).
- The output is an array of `{ path, content }` records. The `path` is the
  destination inside the ZIP the user downloads.

## TemplateContext

| Field | Source | Purpose |
|---|---|---|
| `mcpName` | `config.mcpName` | Package + server name |
| `apiName` / `apiVersion` | `parsedSpec` | Used in README + package description |
| `baseUrl` | `config.baseUrl` | Default `UPSTREAM_BASE_URL` in `.env.example` |
| `upstreamAuth` | `config.upstreamAuth` | Controls auth blocks (apiKey/bearer/none) |
| `mode` | `config.mode` | `'local'` / `'remote'` / `'both'` — drives transports |
| `modeLocalOnly` / `modeHttpOnly` | derived | Convenience flags for `{{#unless}}` blocks |
| `mcpServerToken` | `config.mcpServerToken` | Pre-fills `MCP_SERVER_TOKEN` in `.env.example` |
| `tools[]` | selected endpoints | Each row maps an endpoint to a `server.tool(...)` call |

## Templates

| Template | Emits | Notes |
|---|---|---|
| `package.json.hbs` | `package.json` | Interpolates `mcpName`, `apiName`. Pins SDK versions. |
| `tsconfig.json.hbs` | `tsconfig.json` | Strict, ES2022, NodeNext resolution. `types: ['node']`. |
| `env.example.hbs` | `.env.example` | Conditional sections for `upstreamAuth.type` and `modeLocalOnly`. |
| `gitignore.hbs` | `.gitignore` | Static — node_modules, dist, .env. |
| `http-client.ts.hbs` | `src/http-client.ts` | Single `fetch` wrapper. Injects auth headers per `upstreamAuth.type`. |
| `tools.ts.hbs` | `src/tools.ts` | Iterates `{{#each tools}}`. Each entry calls `server.tool(name, desc, rawShape, async (args) => …)`. The input schema is a **ZodRawShape** (not a `z.object(...)`) — the MCP SDK requires it. |
| `index.ts.hbs` | `src/index.ts` | Wires transports per `mode`. Uses `{{#ifEquals mode "local"}}` blocks. |
| `readme.md.hbs` | `README.md` | Conditional sections for Claude Desktop (stdio) and n8n/Airia (http). |

## Tool name convention

Generated by `toolNameFor(endpoint)` in `mcp-generator.ts`. The rule is
**snake_case of the endpoint label** (`endpoint.label` is the OpenAPI summary
or operationId, normalised by phase 04). Falls back to `'tool'` if the label
sanitises to empty.

Example: `"List products"` → `list_products`, `"Create order"` → `create_order`.

## Input schema generation

`buildTool()` walks `endpoint.params` and emits one entry per param:

```ts
{ id: z.string(), limit: z.number().int().optional() }
```

The actual Zod expression strings come from `buildZodExpression()` in
`zod-schema-builder.ts`. Optional params get `.optional()`; descriptions are
appended via `.describe(...)` only when `config.includeParamDescriptions`
is true.

## How `{{#ifEquals}}` works

Handlebars has no built-in strict-equality block helper. We register one once
at module load (`registerHelpers()`):

```hbs
{{#ifEquals mode "remote"}}…{{/ifEquals}}
```

is equivalent to `mode === 'remote'`. Use it for any discriminated-union
context field (`mode`, `upstreamAuth.type`, etc.).

## Smoke test

`mcp-generator.snapshot.test.ts` runs the generator against
`fixtures/shopify-50.yaml`, drops the bundle in a temp directory, symlinks
the SDK + `@types/node` from the SLICE workspace, then runs
`tsc --noEmit -p tsconfig.json`. This catches template regressions that
typecheck-pass locally but fail in the user's hands.

## Regenerating the bundle

There is no manual step. Bump the templates and the snapshot test re-validates
the full pipeline. Never edit files in `dist/server/server/templates/`
directly — they are copied from `src/server/templates/` by the `copy:templates`
script (invoked by `build:server`).
